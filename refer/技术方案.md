# Obsidian Web Viewer + DOM 抓取 ChatGPT 对话并自动保存 Markdown 的完整技术方案


## 约束与前提

本方案必须把边界先钉死，否则 AI 后面很容易写偏：

1. **仅支持桌面端**。Obsidian 的 Web Viewer 是桌面端核心插件；插件若依赖 NodeJS 或 Electron API，`manifest.json` 应标记 `isDesktopOnly`。([help.obsidian.md](https://help.obsidian.md/plugins/web-viewer), ([Developer Documentation][1]))

2. **仅支持 Web Viewer 路线**，不走官方 API 聊天面板路线。这是你的硬约束。

3. **首版仅支持一个受控 ChatGPT Web Viewer 标签页**。不要一开始支持多标签并发抓取，否则“leaf 与 `<webview>` 对应关系”会显著复杂化。

4. **首版只抓文本和代码块**。图片、表格、附件、引用卡片、复杂富文本先降级处理。

5. **这是高脆弱方案**。Obsidian 官方文档明确说 Web Viewer 不是主浏览器；同时，在 Obsidian 运行期间，第三方插件对 Web Viewer 的 cookie 有完全访问能力。([help.obsidian.md](https://help.obsidian.md/plugins/web-viewer)) 这意味着它能做很多事，也意味着安全风险很高。([Obsidian Help][2])

6. **存在条款风险**。OpenAI 当前服务条款明确禁止自动或程序化提取数据或输出。这个方案适合你本地实验，不适合作为默认公开发布的低风险产品能力。([OpenAI][3])

## 总体目标

插件要完成一条完整的数据链路：

**打开 Web Viewer → 定位目标 ChatGPT 页面 → 注入抓取器 → 拉取快照 → 结构化消息 → 判稳去重 → 写入 Markdown → 保存内部索引 → 页面变化时恢复**

这条链路里，真正最难的是中间 4 步：

* **定位正确的 Web Viewer 实例**
* **稳定提取当前会话消息**
* **流式回复判稳**
* **增量落盘而不是反复全量覆盖**

## 核心设计原则

### 1）把“抓页面”与“写文件”彻底解耦

页面侧脚本只负责返回一个标准 JSON 快照，不直接写文件。
宿主插件只负责拿快照、做状态管理、落盘。

这样后续 DOM 规则变了，只需要改抓取层，不动持久化层。

### 2）宿主“拉取”，而不是 guest“主动推送”

由于你依赖的是 Obsidian 内部 Web Viewer，而不是你自己完全掌控 preload 的 Electron 容器，所以最稳的模式不是复杂的消息桥接，而是：

* 页面侧挂一个全局对象
* 宿主定时 `executeJavaScript()` 调 `collect()`
* 宿主拿到 JSON 后自己处理

Electron 官方文档说明 `<webview>` 是一个 DOM 自定义元素，底层基于 iframe / OOPIF，常用方法和事件都依赖 `dom-ready` 等生命周期；而执行网页上下文脚本通常要等页面准备好。([Electron][4])

### 3）不要把 DOM 规则写死在主流程里

ChatGPT 网页会变，类名会变，层级会变。
所以 DOM 识别规则必须单独做成 **selector profile**，允许多路候选和回退。

### 4）Markdown 是最终资产，JSON 是运行态

用户最终想保留的是 Markdown 会话记录。
但运行时必须保留一份内部状态 JSON，否则无法可靠做“去重、续抓、恢复”。

## 技术架构

建议直接拆成 7 层。

### 第一层：Plugin Entry（插件入口层）

职责：

* 注册命令
* 注册设置页
* 监听 workspace 生命周期
* 启动/停止抓取管理器
* 初始化状态索引

建议入口文件：

```text
src/main.ts
```

入口只做装配，不直接写业务逻辑。

### 第二层：Web Viewer Control（Web Viewer 控制层）

职责：

* 打开 ChatGPT 到 Web Viewer
* 识别当前活动 leaf 是否为目标 Web Viewer
* 维护“受控 leaf”的标识
* 监听 leaf 切换、关闭、布局恢复

Obsidian 官方开发文档说明 workspace/leaf/view 是插件 UI 的基本单元；社区帖子已经验证可以通过 `getLeaf()` + `setViewState({ type: 'webviewer', state: { url, navigate } })` 方式打开 Web Viewer，不过这属于**社区确认的实现方式，不是官方长期稳定契约**。([Developer Documentation][5])

建议封装：

```ts
interface ControlledViewerRef {
  leafId: string;
  expectedUrlPrefix: string;
  createdAt: number;
  lastSeenAt: number;
}
```

### 第三层：WebView Locator（底层 webview 定位层）

职责：

* 从当前 Obsidian 主界面 DOM 中找到对应 `<webview>`
* 建立 “leaf -> HTMLWebViewElement” 的弱映射
* 检查 webview 当前 URL 是否属于 ChatGPT
* 监听目标 webview 生命周期事件

这是整套方案里最脆的一层，因为 Obsidian 官方没有公开一个稳定 API 让你直接拿 Web Viewer 的底层 `<webview>` 元素。你只能通过“workspace 逻辑状态 + 主文档 DOM 扫描”的方式做工程定位。这个结论本身来自官方文档和社区实现方式之间的空缺：官方说能开 Web Viewer，但没有公开说“插件可直接拿其内部 webview handle”；社区实现更多停留在打开视图层面。([Obsidian Help][2])

建议采用双阶段定位：

**阶段 A：逻辑筛选**

* 扫描所有 leaf
* 找 `viewState.type === 'webviewer'`
* 只保留 URL 匹配 `chatgpt.com` / `chat.openai.com` 的候选

**阶段 B：DOM 绑定**

* 在主文档中扫描所有 `<webview>`
* 按“可见性 + 当前 tab 容器 + URL + 最近激活时间”做匹配
* 找到目标实例后缓存引用

建议封装：

```ts
interface WebviewBinding {
  leafId: string;
  webview: Electron.WebviewTag;
  boundAt: number;
  lastUrl: string;
  status: "pending" | "ready" | "lost";
}
```

## 第四层：Capture Runtime（抓取运行时）

职责：

* 等 webview `dom-ready`
* 注入 bootstrap 脚本
* 周期性执行 `collect()`
* 处理注入失效、页面刷新、路由跳转

Electron 官方文档说明 `<webview>` 的方法要等其生命周期准备完成后再用，`dom-ready` 是关键事件；`<webview>` 提供 `executeJavaScript()` 等方法。([Electron][6])

建议事件监听：

* `dom-ready`
* `did-navigate`
* `did-navigate-in-page`
* `destroyed` / 引用失效
* 宿主侧轮询定时器

### 运行模式

1. 找到目标 webview
2. 等 `dom-ready`
3. 注入 bootstrap
4. 启动轮询
5. 每次轮询前先跑 `health()`
6. 若注入对象丢失，则重注入
7. 若 URL 或会话 ID 变化，则切换会话文件

### 推荐轮询参数

* 初始轮询间隔：`1500 ms`
* 空页面 / 登录页：`3000 ms`
* 页面活跃流式输出时：`800~1200 ms`
* 连续失败重试：指数退避，最大 `10000 ms`

## 第五层：DOM Extractor（页面 DOM 抽取层）

这是最关键的业务层。

### 页面侧 bootstrap 设计

注入脚本只挂一个全局对象：

```ts
window.__obsidianChatCapture__ = {
  version: "0.1.0",
  health,
  collect,
};
```

其中：

* `health()`：返回当前页面是否可抓、URL、标题、页面模式
* `collect()`：返回标准化快照

### 标准快照结构

```ts
interface ConversationSnapshot {
  source: "chatgpt-webviewer";
  extractorVersion: string;
  pageUrl: string;
  pageTitle: string;
  capturedAt: string;
  conversationKey: string;
  conversationTitle?: string;
  pageState: "login" | "chat-list" | "conversation" | "unknown";
  messages: ChatMessageSnapshot[];
}

interface ChatMessageSnapshot {
  ordinal: number;
  role: "user" | "assistant" | "system" | "unknown";
  text: string;
  markdownApprox?: string;
  codeBlocks?: Array<{
    language?: string;
    code: string;
  }>;
  rawHtmlSnippet?: string;
  nodeFingerprint?: string;
  hasCompletionActions?: boolean;
}
```

### 会话识别规则

不要依赖单一字段。建议按优先级组合生成 `conversationKey`：

1. URL 中的会话路径 / 会话参数
2. 页面标题
3. 第一条用户消息的规范化文本哈希
4. 首次抓取时间戳兜底

生成规则示例：

```ts
conversationKey = sha256(
  urlConversationPart + "|" +
  normalizedTitle + "|" +
  firstUserMessageHash
)
```

### 消息识别策略

这里不要把技术方案写成某一版 ChatGPT DOM 的“固定类名说明书”。
正确做法是做一个 **SelectorProfile**：

```ts
interface SelectorProfile {
  id: string;
  description: string;
  messageCandidates: string[];
  roleResolvers: Array<(el: Element) => RoleGuess>;
  textResolvers: Array<(el: Element) => string>;
  codeResolvers: Array<(el: Element) => CodeBlock[]>;
}
```

抓取逻辑应该是：

1. 先找“主聊天区域”
2. 再找“消息候选块”
3. 对每个候选块做角色识别
4. 抽文本
5. 抽代码块
6. 生成节点指纹
7. 去噪与排序

### 角色识别规则

按优先级建议：

1. 可访问性属性 / data 属性 / aria 语义
2. 容器结构位置（左/右布局差异）
3. 头像/标识节点的语义文本
4. 与相邻消息的结构关系
5. 无法判断时标为 `unknown`

### 文本抽取规则

建议分层抽：

* 第一层：纯文本
* 第二层：标题、列表、引用转 Markdown 近似
* 第三层：代码块独立抽取
* 第四层：复杂内容保留 `rawHtmlSnippet`

首版目标不是“100% 还原网页渲染”，而是“**语义可读 + 代码不丢**”。

### 代码块抽取规则

对于每个消息块：

* 扫描 `pre`, `code`
* 尝试解析语言标记
* 保留原始换行
* 去除复制按钮等 UI 文本污染

输出结构：

```ts
interface CodeBlock {
  language?: string;
  code: string;
}
```

## 第六层：Snapshot Normalizer（快照归一化层）

页面抓回来的快照不能直接写文件，必须先归一化。

职责：

* 去除空白噪声
* 统一换行符
* 清理零宽字符
* 规整 role
* 计算消息 UID
* 计算消息文本哈希
* 计算整份快照哈希

建议消息 UID 生成：

```ts
uid = sha256(
  ordinal + "|" +
  role + "|" +
  normalizedText + "|" +
  previousMessageUid
)
```

这样即使文本重复出现，也比只 hash 文本更稳。

建议结构：

```ts
interface NormalizedMessage {
  uid: string;
  ordinal: number;
  role: "user" | "assistant" | "system" | "unknown";
  text: string;
  textHash: string;
  codeBlocks: CodeBlock[];
}

interface NormalizedSnapshot {
  conversationKey: string;
  pageUrl: string;
  capturedAt: number;
  pageState: string;
  messages: NormalizedMessage[];
  snapshotHash: string;
}
```

## 第七层：Stability Detector（判稳层）

这是处理 ChatGPT 流式回复的关键。

如果你每次轮询拿到一点新文本就写 Markdown，会导致：

* 文件频繁改写
* 同一条 assistant 消息被拆成很多半成品
* 插件状态混乱

所以必须做 **判稳**。

### 判稳规则

对“最后一条 assistant 消息”做观察：

* 若该消息块已出现复制/点赞/点踩这类回复完成动作，则可直接视为已完成
* 每次轮询计算其 `textHash`
* 若连续 `N` 次不变，则视为稳定
* 或者自首次出现起超过 `T` 毫秒且增长停止，也视为稳定

推荐首版参数：

* `N = 2 或 3`
* `T = 2500~4000 ms`

### 判稳器状态

```ts
interface StabilityState {
  conversationKey: string;
  lastAssistantUid?: string;
  firstSeenAt?: number;
  lastHash?: string;
  stableRepeatCount: number;
}
```

### 处理原则

* **用户消息**：可以立即视为稳定
* **assistant 最后一条消息**：优先看回复完成动作，缺失时再延迟判稳
* **历史消息**：默认视为已稳定

## Markdown 文件格式

建议 **一会话一文件**，不要“一天一个汇总大文件”。

示例：

````md
---
source: chatgpt-webviewer

---

# 会话标题

## User
你好，帮我分析一下……

## Assistant
这里是分析……

```python
print("hello")
````

## User

那如果我换一种方式呢？

````

### Markdown 渲染规则

- `role=user` → `## User`
- `role=assistant` → `## Assistant`
- `role=system` → `## System`
- 文本后跟代码块
- 每条消息之间保留空行

## 内部索引文件

必须有一份内部状态，建议放在：

```text

````

结构建议：

interface SessionIndexEntry {

  filePath: string;
  sourceUrl: string;

## 写文件策略

Obsidian 官方插件指南建议：后台修改文件时，优先使用 `Vault.process`，而不是 `Vault.modify`；Vault API 也提供 `append()` 等方法。([Developer Documentation][7])

所以建议：

* **新建会话文件**：`Vault.create`
* **小幅追加**：可 `append`
* **需要改 frontmatter / 头部信息 / 全量重排时**：`Vault.process`

### 合并算法

每次拿到稳定快照后：

1. 读取该会话的 `SessionIndexEntry`
2. 比较当前 `messages` 与索引中的已落盘消息
3. 找到新增消息尾段
4. 仅把新增部分写入 Markdown
5. 更新 `updated_at`
6. 更新 `message_count`
7. 刷新 `data.json`

## 核心状态机

建议把整个抓取过程做成明确状态机，避免 if/else 失控。

```ts
type RuntimeState =
  | "idle"
  | "openingViewer"
  | "bindingWebview"
  | "injecting"
  | "polling"
  | "saving"
  | "backoff"
  | "error";
```

### 状态转换

* `idle -> openingViewer`
* `openingViewer -> bindingWebview`
* `bindingWebview -> injecting`
* `injecting -> polling`
* `polling -> saving`
* `saving -> polling`
* 任一步失败 -> `backoff`
* 重试成功 -> 对应正常状态
* 连续失败超阈值 -> `error`

## 关键伪代码

### 1）打开 Web Viewer

社区帖子表明可以通过 `leaf.setViewState({ type: 'webviewer', state: { url, navigate } })` 打开 Web Viewer。这个写法可以用，但要在设计里注明：**这是社区确认，不是官方长期 API 承诺**。([Obsidian Forum][8])

```ts
async function openChatGPTInWebViewer(app: App, url: string) {
  const leaf = app.workspace.getLeaf("tab");
  await leaf.setViewState({
    type: "webviewer",
    state: {
      url,
      navigate: true,
    },
    active: true,
  });
  app.workspace.revealLeaf(leaf);
  return leaf;
}
```

### 2）绑定目标 webview

```ts
async function bindWebviewForLeaf(leaf: WorkspaceLeaf): Promise<WebviewBinding | null> {
  const candidates = Array.from(document.querySelectorAll("webview")) as Electron.WebviewTag[];
  for (const webview of candidates) {
    const url = safeGetWebviewUrl(webview);
    if (!url) continue;
    if (!isChatGPTUrl(url)) continue;
    if (!isLikelyBelongToLeaf(webview, leaf)) continue;

    return {
      leafId: getLeafId(leaf),
      webview,
      boundAt: Date.now(),
      lastUrl: url,
      status: "pending",
    };
  }
  return null;
}
```

### 3）注入抓取脚本

```ts
async function ensureBootstrap(binding: WebviewBinding) {
  const webview = binding.webview;

  await webview.executeJavaScript(`
    (() => {
      if (window.__obsidianChatCapture__) return true;

      function health() {
        return {
          ok: true,
          url: location.href,
          title: document.title,
        };
      }

      function collect() {
        // 调 dom-extractor 的打包逻辑
        return window.__OBSIDIAN_CAPTURE_COLLECT__?.();
      }

      window.__obsidianChatCapture__ = {
        version: "0.1.0",
        health,
        collect
      };

      return true;
    })()
  `);
}
```

### 4）轮询抓取

```ts
async function poll(binding: WebviewBinding) {
  const webview = binding.webview;

  const snapshot = await webview.executeJavaScript(`
    window.__obsidianChatCapture__?.collect?.() ?? null
  `);

  if (!snapshot) return;

  const normalized = normalizeSnapshot(snapshot);
  const result = stabilityDetector.accept(normalized);

  if (!result.readyToPersist) return;

  await markdownWriter.mergeAndSave(normalized);
  await sessionIndex.update(normalized);
}
```

## 异常与恢复设计

### 场景 1：页面刷新后注入对象消失

处理：

* 每轮轮询前先调用 `health()`
* 若不存在 `window.__obsidianChatCapture__`，重注入

### 场景 2：用户切换到另一个会话

处理：

* 比较 `pageUrl` 与 `conversationKey`
* 若变化，则查找或新建对应 Markdown 文件
* 原会话状态挂起，新会话开始轮询

### 场景 3：用户退出登录或停在登录页

处理：

* `pageState = login`
* 暂停抓取正文
* 降低轮询频率
* 状态栏提示“等待登录完成”

### 场景 4：DOM 规则失效，抓取为空

处理：

* 连续 `N` 次空抓取不清空已知状态
* 输出调试日志和当前页面片段
* 提供命令手动导出页面 HTML 片段用于修复 selector

### 场景 5：插件重载或 Obsidian 重启

处理：

* 读取 `data.json`
* 若存在 `controlledViewer`
* 在 layout ready 后尝试重新定位该 leaf / webview
* 成功则恢复轮询

## 调试与日志设计

调试能力一定要有，否则 AI 写出来以后你几乎无法排查。

建议日志分 3 级：

* `info`：绑定、注入、保存成功
* `warn`：暂时抓不到、会话切换、空快照
* `error`：executeJavaScript 失败、写文件失败、状态损坏

建议 debug 目录输出：

* 最近一次原始快照 JSON
* 最近一次归一化快照 JSON
* 最近一次失败时的 URL / title / pageState
* selector 命中统计

## 对 AI 编码的明确要求

你后面把这个方案喂给 AI 时，最好附上这些硬要求，不然它很容易偷懒写成脆弱代码：

1. **不要把 DOM selector 写死在主流程中**，必须单独抽成 `selector-profiles.ts`。
2. **不要直接在抓取脚本中写 Obsidian 文件**。
3. **所有写文件操作必须经过 persistence 层**。
4. **后台改文件优先用 `Vault.process`**。([Developer Documentation][7])
5. **必须实现 session index**，不能只靠 Markdown 内容反推状态。
6. **必须实现流式回复判稳**，否则自动保存会很差。
7. **必须保留 debug 开关**，能导出原始快照。
8. **必须标记 `isDesktopOnly`**，因为方案依赖 Node/Electron / Web Viewer。([Developer Documentation][1])

## 最终落地判断

这套方案的定位很明确：

* **能做**
* **适合桌面端个人工具**
* **首版开发量中等**
* **后续维护成本高**
* **脆弱点主要在 Web Viewer 定位和 ChatGPT DOM 变化**
* **安全与条款风险要自担** ([Obsidian Help][2])
